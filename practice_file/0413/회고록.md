## 🦝 멀티 에이전트 설계 (The SskkaMu Team)

에이전트를 **기획자, 개발자, 검수자** 세 명으로 나누는 것이 가장 효율적입니다.

### 1. 분석 에이전트 (Planner Agent)
* **역할:** 사용자의 모호한 입력을 분석하여 의도(분류)를 파악합니다.
* **임무:** "수아가 글 썼어" → `INSERT` 작업이며, 작성자는 `수아`, 내용은 `...` 임을 추출합니다.
* **출력:** 정형화된 JSON 데이터 (예: `{"action": "insert", "data": {...}}`)

### 2. SQL 생성 에이전트 (SQL Writer Agent)
* **역할:** 분석된 JSON 데이터를 받아 실제 MariaDB 문법에 맞는 SQL을 생성합니다.
* **임무:** 오직 SQL 쿼리만 생성하며, 테이블 스키마 정보를 완벽하게 숙지합니다.
* **출력:** `USE sooah_board; INSERT INTO ...;`

### 3. 검수 에이전트 (Reviewer Agent)
* **역할:** 생성된 SQL이 위험하지 않은지(예: `DROP TABLE` 등), 문법이 맞는지 검사합니다.
* **임무:** 보안 위협 요소가 있다면 차단하고, 정상적이라면 실행 에이전트에게 전달합니다.
* **출력:** `Pass` 또는 `Fail (사유)`

## 🛠️ 예상 코드 구조 (Python)

에이전트별로 `System Prompt`를 다르게 설정한 `llm` 객체를 여러 개 만들어 연결합니다.

```python
# 에이전트별 전담 LLM 설정
planner = ChatOllama(model="gemma2:2b", system="사용자 의도를 JSON으로 분석해라구리.")
sql_writer = ChatOllama(model="gemma2:2b", system="JSON을 받아서 MariaDB 쿼리만 써라구리.")
reviewer = ChatOllama(model="gemma2:2b", system="SQL 보안 위험을 체크해라구리.")

@router.post("/insert")
async def multi_agent_insert(request: UserInput):
    # Step 1: 의도 분석
    plan_res = planner.invoke(request.user_input)
    
    # Step 2: SQL 생성
    sql_res = sql_writer.invoke(plan_res.content)
    
    # Step 3: 보안 검수
    review_res = reviewer.invoke(sql_res.content)
    
    if "PASS" in review_res.content.upper():
        execute_sql_from_llm(sql_res.content)
        return {"status": True, "message": "멀티 에이전트 협업 성공이다구리!"}
    else:
        return {"status": False, "message": "검수 에이전트가 거절했다구리!"}
```
